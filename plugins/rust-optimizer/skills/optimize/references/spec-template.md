# `OPTIMIZATION_SPEC.md` — the autopilot handoff contract

This is the interface between the optimizer and autopilot. `autopilot:plan` ingests it, and because
every finding's Definition of Done is machine-checkable in autopilot's own vocabulary, it plans with a
high readiness score and no enrichment. Emit the file at the repo root (or wherever the user will feed
`autopilot:plan`).

## Definition-of-Done vocabulary

Each `definition_of_done` line is one of:

- `cmd: <shell command>` — passes when the command exits `0`. Use for anything a tool can prove
  (`cmd: cargo machete`, `cmd: cargo audit`, `cmd: cargo build --workspace`).
- `grep: <pattern> in <glob>` — passes when the pattern **is present** (the fix added something).
- `grep:absent: <pattern> in <glob>` — passes when the pattern **is gone** (the fix removed a smell).

Prefer `cmd:` when a tool can settle it — an executable check can't be faked green. Use `grep:` for
structural changes a command can't easily assert.

## Per-finding section structure

Use this exact structure, one section per finding:

```markdown
### <id> — <short title>

- **severity:** critical | high | medium | low
- **evidence:** <file:line>, <file:line>
- **deliverable:** <what the fix produces — described, not applied>
- **definition_of_done:**
  - cmd: <command that must exit 0>
  - grep: <pattern> in <glob>
  - grep:absent: <pattern> in <glob>
- **validation:** <how a human confirms it empirically — e.g. a push:false smoke build>
- **est_impact:** <deterministic and/or empirical, warm/cold labeled — see metrics.md>
- **risk:** none | low | medium | high
- **depends_on:** [<id>, ...]
```

## File-level trailers

After the findings, add:

- **`suggested_sequence:`** — an ordered list of ids: quick wins first, then tradeoff decisions.
  Respect `depends_on` (e.g. remove the disk hack B1 only _after_ A1/A3).
- **`risk_phases:`** — the ids that need a human checkpoint even under autopilot (release/arm64
  changes, security-gate enforcement). These map to autopilot's `risk_phases:`.

## Worked example (excerpt)

```markdown
# OPTIMIZATION_SPEC.md

> Generated by rust-optimizer. Two human checkpoints: (1) curate these findings, (2) approve the
> autopilot plan. Recommended first run: `autonomy: reviewed`.

### A1 — Build workspace once, share via nextest archive

- **severity:** high
- **evidence:** .github/workflows/ci.yml:41, .github/workflows/ci.yml:78, .github/workflows/coverage.yml:22
- **deliverable:** An archive job (`cargo nextest archive`) whose output the test/coverage jobs consume.
- **definition_of_done:**
  - grep: nextest archive in .github/workflows/**
  - grep: archive-file in .github/workflows/**
- **validation:** CI green; test jobs no longer run `cargo build`/`cargo test` from scratch.
- **est_impact:** deterministic: full-workspace-compiles-per-PR 4 → 2. empirical: warm time-to-green −X min (≥N runs each side).
- **risk:** low
- **depends_on:** []

### E1 — Enforce cargo audit (remove `|| echo` theater)

- **severity:** high
- **evidence:** .github/workflows/ci.yml:96
- **deliverable:** The audit step blocks on non-zero instead of swallowing the failure.
- **definition_of_done:**
  - cmd: cargo audit
  - grep:absent: || echo in .github/workflows/**
  - grep:absent: continue-on-error: ?true in .github/workflows/**
- **validation:** Introduce a known-vulnerable dep on a scratch branch; CI must fail.
- **est_impact:** deterministic: enforced-gates 0 → 1.
- **risk:** medium
- **depends_on:** []

### D1 — Native arm64 release split (replace QEMU)

- **severity:** high
- **evidence:** .github/workflows/release.yml:33 (setup-qemu-action), :51 (platforms: linux/arm64)
- **deliverable:** arm64 built on `${{ vars.HEAVY_RUNNER || 'ubuntu-latest' }}` mapped to
  `ubuntu-24.04-arm`, push-by-digest, `docker buildx imagetools create` manifest merge.
- **definition_of_done:**
  - grep:absent: setup-qemu-action in .github/workflows/release.yml
  - grep: ubuntu-24.04-arm in .github/workflows/**
  - grep: buildx imagetools create in .github/workflows/**
- **validation:** `push: false` smoke build on both arches first; compare build time vs QEMU baseline.
- **est_impact:** empirical: arm64 image build ~59m → ~5m (native, public repo).
- **risk:** high
- **depends_on:** []

---

suggested_sequence: [A3, A1, E1, F1, D1]
risk_phases: [E1, D1]
```

Note the `> Generated by rust-optimizer …` blockquote at the top — it reminds the human of the two
checkpoints and the recommended `reviewed` autonomy for the first pass.
